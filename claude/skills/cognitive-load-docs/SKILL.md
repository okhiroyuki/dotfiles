# ドキュメントの認知負荷設計

ドキュメント全体の情報構造を、読み手のワーキングメモリにかかる負荷の観点で設計・レビューする。

対象は構造(何をどの順で、どこまで詳しく出すか)であり、文の書き方(語調・修辞)は扱わない。日本語の文単位の規範は japanese-tech-writing スキルに委ねる。

## 判断軸: 負荷の3分類

Sweller の認知負荷理論に基づき、負荷を3種類に分けて考える。減らすべきは Extraneous のみで、Intrinsic は削れず、Germane は減らしてはならない良い負荷である。

- **Intrinsic Load**（内在的負荷）: 内容そのものの難しさ。設計をどう変えても減らせない。
- **Extraneous Load**（外在的負荷）: 構成の悪さによる無駄な負荷。文脈なしの具体例、迷子になる構成、異なる文脈の混在などが原因。**これを削るのが設計の主目標**。
- **Germane Load**（学習的負荷）: 読み手が頭の中に「地図」（スキーマ）を組み立てるために使う、必要な負荷。減らしすぎると逆に理解が浅くなる。

構成上の問題を見つけたら、それが Extraneous（削れる無駄）なのか Germane（必要な負荷で、削ると理解を損なう）なのかをまず切り分ける。

## 手順

### 1. ドキュメント種別を1つに定める（Diátaxis）

異なる文脈の情報を1つの文書に混ぜると、読み手はモードの切り替えを強いられ、認知負荷が跳ね上がる。書く前に、または既存文書をレビューするときは、以下の4種のうちどれに当たるかを確認する。

- **Tutorial**: 初めての読み手を手を引いて成功体験に導く
- **How-to Guide**: 既にわかっている読み手が特定の目的を達成する手順
- **Reference**: 事実の正確な記述。解釈や誘導を含まない
- **Explanation**: 背景・設計判断の「なぜ」を説明する

複数の種別が1文書に混在している場合（例: 手順の合間に設計思想の長い説明が挟まる）、分割を提案する。README のように複数種別を1ファイルに収めざるを得ない場合は、セクションごとに種別を意識して切り替え、混ぜない。

### 2. 冒頭に地図を置く（Progressive Disclosure）

地図（全体像）がない状態でいきなり詳細や具体例に入ると、読み手は文脈が来るまで未解決の情報をワーキングメモリに保持し続けねばならず、オーバーフローする。

- 冒頭で「これは何か／何ができるか」を1〜2文で述べてから詳細に入る
- 各セクションも「ミニ概要 → 詳細」の入れ子構造を繰り返す
- 導入時が最も負荷が高いため、「まずこれをやれば動く」という最小手順を早い位置に置く

### 3. チャンク数を確認する（Miller's Law）

ワーキングメモリが同時に保持できる項目は3〜4個程度とされる。1セクションの主要ポイントがそれを超えると、読み手は前の項目を覚えていられなくなる。

- 1セクションの主要ポイントは3〜4個までを目安にする
- 箇条書きが5項目以上になったら、共通性でグルーピングし直せないか検討する

### 4. ルール・手順にWHYを添える

理由のない指示は記憶に定着しにくい。ルールや制約を書くときは、なぜそうするのかを一言添える（「〜する。理由: 〜のため」の形でよい）。

### 5. その他の実務チェック

- 同じ概念に複数の呼び方を当てていないか（一貫した用語）
- 新しい概念を、読み手が既に知っている概念に橋渡ししてから導入しているか
- 専門用語・社内用語は初出時に説明を添えているか

## 成功基準

レビュー・執筆の完了前に以下を自己点検する。

- [ ] ドキュメントの種別（Tutorial/How-to/Reference/Explanation）が単一に定まっている、または種別ごとにセクションが明確に分離されている
- [ ] 冒頭に地図（概要）があり、詳細より先に読める
- [ ] 各セクションの主要ポイントが3〜4個以内、超える場合はグルーピングされている
- [ ] ルール・制約にWHYが添えられている
- [ ] 同一概念の呼び方が文書全体で揺れていない
