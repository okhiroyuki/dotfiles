## 1. スキルを作るべきかの判断

スキルを新規作成する前に以下をすべて確認する：

- 同じタスクで同じ文脈・指示を2回以上繰り返し手入力したか？
- 今後も継続して発生する見込みがあるか？
- カスタムインストラクション（CLAUDE.md）では足りないか？

いずれかがNoなら作らない。投機的に作ると保守負債が増えるだけ。

## 2. frontmatter の必須項目

| フィールド    | 必須度 | ルール                                                                         |
| ------------- | ------ | ------------------------------------------------------------------------------ |
| name          | MUST   | 64文字以内・小文字・ハイフンのみ・予約語(anthropic/claude)不可                 |
| description   | MUST   | 1024文字以内・三人称・具体動詞・ユースケース・境界(「〜には使わない」)を含める |
| model         | MUST   | 下記のmodel選定フローに従う                                                    |
| argument-hint | 推奨   | 引数がある場合のみ                                                             |

### model 選定フロー

model は MUST で明示する。未指定のまま放置しないこと。
既存スキルを編集する際、model が未指定なら適切な値を補うこと。

- **sonnet**（デフォルト推奨）: 定型的・反復的タスク、テンプレート処理、スニペット生成
- **opus**: 技術判断・推論・比較検討・文章生成を含むタスク

迷ったら opus。判断基準に迷うコスト > opus の実行コスト。
既存スキルが `model: sonnet` で、技術判断・推論・文章生成を含む場合は opus に見直すこと。

### description のチェックポイント

- 三人称で書かれているか（"This skill extracts..." ではなく "Extracts..."）
- 具体的な動詞が含まれているか（"helps with" は不可、"extracts/merges/validates" は可）
- 適用対象と非適用対象の両方が書かれているか
- キーワードマッチではなく意味的理解でトリガされることを意識しているか

## 3. スキル設計時のコンテキスト配慮

- SKILL.md は **100行以下**を維持する（公式上限は500行だが厳格に抑える）
- 重い処理・長文は `scripts/`（コード）や `prompts/`（プロンプトテンプレート）に切り出し、本文では参照する
- 参照ファイルへのパスはフォワードスラッシュ（/）を使用する
- 参照の深さは1階層まで（SKILL.md → 参照ファイルの先にさらに参照を作らない）
- スキル本文には「成功基準」をMUSTで含める。Claudeが自己チェックできるようにする
- 重要なルールには "MUST" / "SHOULD" / "REQUIRED" などの強い表現を使う
- 時間敏感な情報は避ける。避けられない場合は「古いパターン」セクションを設ける
- 巨大スキルはマイクロスキルに分割すること

## 4. 評価プロセス（Evaluation-Driven Development）

スキルを作る前に以下のプロセスを経ること：

1. **ギャップ特定**: スキルなしで代表タスクを実行し、失敗・不足を記録
2. **ベースライン測定**: スキルなしでの性能を記録
3. **最小限の指示を書く**: ギャップを埋めるのに十分な内容のみ
4. **テスト**: 通常操作・エッジケース・スコープ外の3シナリオで検証
5. **反復**: ベースラインと比較しながら洗練

ドキュメントを書く前に評価を作る。想像上の問題ではなく、実際の問題を解く。

## 5. 既存スキル編集時のチェック

**frontmatter**

- model が未指定なら適切な値を補う
- `model: sonnet` でも技術判断・推論・文章生成を含むなら opus に上げ直す
- description が三人称・具体動詞・境界を含むか再確認
- name が予約語(anthropic/claude)を含まないか

**本文**

- SKILL.md が100行を超えていないか（超なら別ファイルへ切り出し）
- 参照ファイルへのパスがフォワードスラッシュか
- 時間敏感な情報が古くなっていないか

**構成**

- 使わなくなったスキルは廃止
- 巨大スキルならマイクロスキルへの分割を検討

## 6. subagent 委譲の設計

複数ステップ・長文出力が想定される処理は subagent に委譲する設計にする。

- subagent に切り出すことで、メインコンテキストの圧迫を防ぎ、出力結果のみを受け取る
- 活用基準: 中間結果が大量に出る、深い検索・ログ分析が必要、メイン会話の流れを遮りたくない場合
- 隔離したいサイドタスクはsubagentで実行し、要約のみ返却させる

## 7. 最終防衛策

上記を励行してもコンテキストが溢れる・モデルが混乱する事象が発生した場合は、`/clear` で対処する。
